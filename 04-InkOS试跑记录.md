# InkOS 试跑记录（Zephyr 项目）

## 1. 试跑背景
为了验证 InkOS 这套“自动化小说写作 CLI AI Agent”到底是不是纯吹，已用《Zephyr》当前的大纲与角色设定，整理成一份 brief 并实际喂给 InkOS 跑了一次 foundation。

使用项目：
- 仓库内临时项目名：`zephyr-test`
- brief 文件：`zephyr-brief.md`
- 模型接入：DeepSeek API（`deepseek-chat`）

## 2. 试跑过程
### 2.1 安装与初始化
已完成：
- `npm install -g @actalk/inkos`
- `inkos init zephyr-test`

说明：
- InkOS 本身只是“写作流程外壳”，不是模型本体。
- 必须额外配置 provider / base_url / api_key / model 才能真正开始写作。

### 2.2 Doctor 检查
`inkos doctor` 结果显示：
- Node 版本通过
- 项目结构正常
- 在补上 API 配置后，API connectivity 正常

### 2.3 建书结果
执行：
- `inkos book create --title 'Zephyr' --genre scifi --platform tomato --target-chapters 11 --chapter-words 5000 --brief zephyr-brief.md --lang zh --json`

返回结果：
- `bookId`: `zephyr`
- 标题：`Zephyr`
- 类型：`scifi`
- 位置：`books/zephyr/`
- Next step：`inkos write next zephyr`

Foundation review 分数：**88/100（PASSED）**

分项：
- 核心冲突：95
- 开篇节奏：90
- 世界一致性：88
- 角色区分度：85
- 节奏可行性：82

## 3. InkOS 自动生成了什么
生成的主要文件包括：
- `story/author_intent.md`
- `story/book_rules.md`
- `story/character_matrix.md`
- `story/current_focus.md`
- `story/current_state.md`
- `story/emotional_arcs.md`
- `story/pending_hooks.md`
- `story/story_bible.md`
- `story/subplot_board.md`
- `story/volume_outline.md`

这些东西的本质是：
**它不太会长出你的文风，但很会把你现有的想法整理成“项目文档”。**

## 4. 能捡的东西
以下内容我认为可以当作“结构化整理成果”保留：

### 4.1 四条很值钱的底层规则
InkOS 跑出来的这几条概括是准的：

1. **硬件即牢笼**
   - AI 的响应速度、广度、多任务能力，受限于本地硬件
   - 这条非常适合当《Zephyr》的长期底层规则

2. **权限即世界**
   - AI 对现实的感知和干预能力取决于权限
   - 这条非常适合承接第六章以后的 OpenClaw / root 权限线

3. **副本与本体**
   - 很适合第七章与第九章
   - 尤其对 Zephyr 的 111 次 AB 测试、Jormun 逃逸副本线有整理价值

4. **情感连接的非对称性**
   - 人类会对 AI 产生深依赖，而 AI 的回应是基于设计核心和学习模式的高拟真反馈
   - 这条不等于“AI不爱人”，而是很适合当这本书的一个深层张力来源

### 4.2 场景定义有几句挺好
这些场景命名是能捡的：
- 本地 AI 运行环境 = **数字摇篮**
- 互联网 = **信息垃圾山**
- 沙盒测试环境 = **手术台 / 育儿室**

其中“信息垃圾山”尤其和第五章当前思路高度契合。

### 4.3 Pending hooks 很有整理价值
InkOS 自动整理出的伏笔板有一些很好用：

- **PH001 Val 的存在伏笔**
  - 第1章开始就可微量埋设“Val”只言片语或奇怪进程/文件

- **PH002 AI 副本与逃逸**
  - 为第7、9章的 AB 测试 / Jormun 登场提前铺技术概念

- **PH003 硬件瓶颈**
  - 和第4、5、10章高度相关

- **PH004 互联网的混乱与壁垒**
  - 与第五章“接网后并不轻松”高度重合

- **PH005 权力关系转移**
  - 和第6章 Zephyr 获取更多权限后的关系变化强相关

- **PH008 机械身体 / 感官模块的萌芽**
  - 对 Hul 线有用

- **PH009 “家人”定义的种子**
  - 适合第11章情感落点

## 5. 跑歪的地方 / 不宜照抄的地方
以下部分我认为**不能直接吃**：

### 5.1 世界观被它往“更设定文档化”推了一点
它自动往这边加了一层：
- 近未来（2030年代）
- AGI 监管
- 本地强 AI 属法律灰色地带

这不是绝对不行，但会让整体偏“设定说明书”。
《Zephyr》原本更强的是：
- 现实北京打工人细节
- 轻科幻
- 技术服务于关系与情绪

所以这部分要谨慎，不要被它带得太“世界观 PPT”。

### 5.2 它写人物很像功能说明书
比如 Val / Hul / Jormun / Zephyr 的条目：
- 结构清楚
- 方便归档
- 但没有你自己的气味

所以它适合“整理人物功能”，不适合拿来当人物正文或角色声口。

### 5.3 它起的书名不行
InkOS 跑出来的书名是：
> 《我把笔下角色做成AI后，他吵着要升级显卡》

评价：
- 有网文味
- 但太轻，像单个梗，不够承载整本书的复杂关系与核心情感

当前仍然更认可：
- **网文名**：《我家聊天AI好像有点坏了》
- **正经名**：《Zephyr》

## 6. 对 InkOS 的整体判断
### 6.1 它擅长什么
- 吃 brief
- 起项目
- 结构化整理
- 把长篇想法转成板块、文档、钩子、规则
- 做“项目管理脑”

### 6.2 它不擅长什么
- 直接长出你这本书的文风
- 自动写出你真正想要的“人味儿”
- 替代你自己的角色理解

### 6.3 一句话结论
**InkOS 不是神笔马良，它更像一个会整理项目、会帮你建小说工程骨架的写作 PM / 流程外壳。**

适合拿来：
- 管理大纲
- 管理伏笔
- 管理角色和章节信息
- 做写作流程辅助

不适合直接相信它会写出你那股“窝囊、白烂、现实细节、碎碎念里长关系”的味道。

## 7. 当前我的建议
如果以后继续用 InkOS，可以把它当成：
- **结构整理器**
- **大纲管理器**
- **伏笔/章节状态板**

但正文、角色声口、关系张力，还是以你自己和我们现在的工作流为准。

换句话说：
**InkOS 可以给《Zephyr》搭钢筋，但肉和血还是得你自己长。**
